# TXT to Issue

テキストファイルを使ってGitHubのIssueを一括管理するためのシステムです。
わざわざブラウザでGitHubを開かなくても、手元のエディタで箇条書きを作る感覚でIssueの作成や完了（クローズ）ができます。
これを導入することによって、リポジトリを作り直したとしてもローカルのgitにIssueを記録するため、草として記録を残すことができます。

---

## セットアップ手順

このシステムを動かすために、リポジトリ内に以下のファイル群を追加してください。

```text
project_root/
├── issues.txt             # Issueの中身を書き込むメインのファイル
└── .github/
    └── scripts/
        ├── sync_issues.py # Issueの作成やクローズを判断するPythonスクリプト
        └── issues.yml     # ファイルを更新したときにスクリプトを自動で動かすための設定ファイル
```

---

## 書き方

アイデアやタスクを登録したいときは、issues.txt を以下のように記述します。

* **1行目**: Issueのタイトル
* **2行目以降**: Issueの本文（メモや詳細）
* **ブロックの区切り**: 次のIssueを書く前に、必ず空行（改行2回）を入れます。

記述が終わったら、通常通り git add / commit / push を行ってください。
GitHub側で自動的に中身が読み取られ、Issueが作成されます。

---

## Issueを完了（クローズ）させる方法

すでに作成したIssueを終わらせたいときは、issues.txt 内のタイトルの末尾に 半角スペース ＋ closed を追記してプッシュしてください。
書き方の例はこのリポジトリのissues.txtを参照してください。

---

## このシステムの特長

* **二重投稿の防止**: すでに同じタイトルのIssueが存在する場合は、自動的にスキップされます。プッシュするたびに同じIssueが増え続けることはありません。
* **コンフリクト（衝突）の回避**: このシステムは、GitHub側からあなたの手元のファイルを書き換えることはありません。そのため、git pull を忘れて作業を始めても、Gitのエラーが起きにくい設計になっています。
* **並列処理の安全化**: 連続でプッシュしても、順番に1つずつ処理されるようになっているため、動作が安定しています。

---

## 注意事項
* **タイトルの変更**: Issueが作成された後にタイトルの1行目を書き換えてしまうと、システムが「新しい別のIssue」だと判断して新しく作成してしまいます。タイトルを修正したい場合はGitHubの画面上で行うか、重複を許容した上で修正してください。
* **本文のみの更新**: 本文だけを書き換えてプッシュしても、タイトルが変わっていない限り、GitHub上のIssueの内容は更新されません（スキップされます）。現状は「新規作成」と「クローズ」に特化したツールです。
